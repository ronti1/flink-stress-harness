Flink Stress Harness

A configurable, synthetic-data Apache Flink streaming job whose one purpose is to prove a new Kubernetes cluster behaves the same as an old one before you migrate real workloads. Run the identical job on both, compare in Grafana, and let the scorecard give you a machine-readable PASS/FAIL.

Flink 1.18 · Java 11 K8s Operator 1.12 80 automated tests Docker · Helm · Prometheus · Grafana

This page is a standalone handover guide. The same content (and more code detail) lives in README.md; the deep maintainer/agent guide is .claude/CLAUDE.md.

Overview

You run the exact same job on the old and new cluster with identical config and a fixed RNG seed, so the only variable is the infrastructure. The job pushes metrics into Prometheus; Grafana visualizes them; a Python scorecard compares the two clusters and exits non-zero on regression (CI-gate friendly).

Everything is a knob

Rate & pattern, record size, key skew, late data, event/processing time, window type, UDF weight, state backend, checkpointing, restart, and fault injection at 5 points.

Runs on your laptop

Docker Compose brings up Flink + Prometheus + Grafana + MinIO — no Kubernetes needed to see it work.

Ships to clusters

A Helm chart deploys the job as a FlinkDeployment plus a self-contained observability stack.

Objective verdict

The scorecard turns "looks about the same" into a per-metric PASS/FAIL with tunable tolerances.

In one sentence
A Flink job is a graph of operators; the JobManager schedules them onto TaskManager slots; windows group records; checkpoints make it fault-tolerant. Everything else is a knob.

The pipeline

A single linear pipeline. Faults can be injected at five points — three inside real operators (SOURCE, AGGREGATE, SINK) and two as pass-through operators at the boundaries (AFTER_SOURCE, BEFORE_SINK). A point appears in the graph only if you target it.

          SOURCE        AFTER_SOURCE      AGGREGATE       BEFORE_SINK     SINK   <- optional fault stages
          (in-op)       (boundary map)    (in-op)         (boundary map)  (in-op)
             |             |                  |                |             |
 SyntheticSource ─► [watermarks] ─► keyBy ─► window + aggregate ─► sink
                  (event-time only)                                  (metrics | console)

 Flink PrometheusReporter ─► Prometheus ─► Grafana dashboard
 scorecard/compare.py ─────► queries BOTH clusters' Prometheus ─► PASS/FAIL + exit code
  1. SyntheticSource — generates records at the configured rate/pattern/size, with key-skew and late-data injection. A seed makes the stream byte-identical on both clusters.
  2. watermarks — assigned only for event-time time/session windows.
  3. keyBy — partition by key (KEY_BY) or round-robin (REBALANCE).
  4. window + aggregate — groups records; the aggregator's CPU cost and per-key state size are independent knobs.
  5. sink — emits Flink metrics (throughput + latency histogram) or logs to console.
Latency definition
e2eLatencyMs = (sink wall-clock) − (newest source emit-time in the window). It includes window buffering by design — a relative A/B signal, identical on both clusters, not an SLA.

Fault injection

Two kinds of fault, each targetable to any of five stages:

fault.app

An application exception (HarnessInjectedException) → Flink region failover / restart governed by restart.strategy. Tests recovery from transient code errors.

fault.sys

A system halt: Runtime.halt() abruptly kills the TaskManager JVM (no shutdown hooks, no K8s RBAC needed) — closest thing to "a node died". Tests TM-loss recovery + state restore.

Each fault independently picks:

KeyValues / defaultMeaning
fault.{app,sys}.enabledfalseTurn the fault on.
fault.{app,sys}.stageSOURCE·AFTER_SOURCE·AGGREGATE·BEFORE_SINK·SINK (def AFTER_SOURCE)Where — in-operator vs boundary.
fault.{app,sys}.triggerModeEVERY_N_RECORDS · EVERY_MSHow often measured.
fault.{app,sys}.triggerN60000 app / 120000 sysRecords or ms between triggers.
fault.{app,sys}.maxOccurrences1Cap per subtask; ≤0 = unlimited.
fault.{app,sys}.probability1.0Probability of firing once due.
fault.sys.haltExitCode1JVM exit code for the halt.

All logic lives in one place — FaultInjector, a pure, unit-tested scheduler. The boundary operator FaultInjectionMap and the three in-operator hooks all delegate to it, so behaviour is identical wherever you inject.

Limitation
fault.app is always an exception and fault.sys is always a halt, so at most one of each kind is active at once (each on its own stage). To inject two exceptions at two stages you'd generalize the config — see Extending.
Recovery testing tip
The synthetic source has no checkpointed offsets — on restart it regenerates from scratch, so an EVERY_N_RECORDS fault re-fires at the same count after every restart (infinite loop unless maxOccurrences caps it). For recovery scenarios use EVERY_MS + a small maxOccurrences, and enable checkpointing.

Time & watermarks

The part newcomers find hardest — exactly what the knobs do:

  • EVENT_TIME (default): windows assigned by each record's event time; Flink needs watermarks to know a window is complete. The harness assigns timestamps + a watermark generator before keyBy.
  • PROCESSING_TIME: windows assigned by wall-clock arrival. No watermarks, no late data — those knobs go inert. Fires on wall-clock timers, which behave very differently under backpressure (a great A/B axis).

Watermark strategies (event-time only): BOUNDED_OUT_OF_ORDERNESS (allow a bounded lateness — the usual choice), MONOTONOUS (timestamps never go backwards — tightest), NONE (no watermarks; event-time windows fire only at end-of-stream).

Fair A/B
Set boundedOutOfOrdernessMs ≥ late.maxMs so injected late data isn't silently dropped (the late-data scenario does this).

Deploying to Kubernetes (Helm)

Prerequisite
The Flink Kubernetes Operator 1.12 must already be installed cluster-wide on each target. This chart ships only the FlinkDeployment CR + a self-contained Prometheus + Grafana.
# Build & push the image to a registry both clusters can pull, then on EACH cluster:
helm install harness helm/flink-stress-harness \
  --set image.repository=<registry>/flink-stress-harness --set image.tag=<tag> \
  --set harness.scenario=latency-soak \
  --set state.backend=rocksdb \
  --set checkpoint.enabled=true --set checkpoint.dir=s3://flink-checkpoints/harness \
  --set s3.endpoint=http://minio.minio.svc:9000 \
  --set s3.accessKey=<key> --set s3.secretKey=<secret>

Use identical values on both clusters. Lint/render without a cluster:

docker run --rm -v ${PWD}:/apps -w /apps alpine/helm:3.14.4 lint helm/flink-stress-harness
docker run --rm -v ${PWD}:/apps -w /apps alpine/helm:3.14.4 template t helm/flink-stress-harness
Why flink-rbac.yaml?
In application mode the JobManager is a Kubernetes API client — at runtime it creates TaskManager pods, a ConfigMap (HA/leader election) and a Service, so it needs a ServiceAccount + Role. This is standard K8s RBAC and applies to on-prem clusters exactly as to cloud. The Flink Operator's own chart often also provisions a flink SA; if so this template may be redundant — point the CR at the operator's SA and disable it.

The A/B scorecard

scorecard/compare.py queries both clusters' Prometheus over a window, aggregates signal metrics, asserts the candidate is within tolerance of the baseline, prints a table, and exits non-zero on regression.

python scorecard/compare.py \
  --baseline  http://<clusterA-prometheus>:9090 \
  --candidate http://<clusterB-prometheus>:9090 \
  --tolerances scorecard/tolerances.yaml --window 10m

Metrics, aggregation, direction (higher/lower-is-better) and per-metric tolerance are declared in scorecard/tolerances.yaml — edit that, no code change.

Gotcha — wastes hours
Flink's Prometheus reporter sanitizes label values: spaces and - become _, and operators get a Sink:/Source: prefix (your latency-sink appears as operator_name="Sink:_latency_sink"). Always use regex selectors (=~".*latency.sink.*"), never exact names. The dashboard and scorecard already do this.

Extending the harness

ChangeWhere
Add a config knobField + p.get… in HarnessConfig; bounds-check in validate(); test in HarnessConfigTest.
Add a window typeWindowType enum + case in WindowFactory; IT in HarnessJobBehaviorITCase.
Add a rate patternRateMode + RateController (pure fn; test in RateControllerTest).
Add an in-operator fault stageFaultStage + call FaultInjector.forStage in that operator.
Two exceptions/halts at onceGeneralize fault.app/fault.sys into a List<FaultConfig>.
Add a scorecard metricEntry in scorecard/tolerances.yaml (no code).
House rules
Target Flink 1.18 exactly (open(Configuration), not open(OpenContext)); behavioural knobs flow as job args, not cluster config; keep the testability seams pure (FaultInjector.evaluate, LatencyMeasuringSink.latencyMs); no Flink-internal classes. Update all three docs when knobs change.

What is actually verified

  • Automated (80 tests): rate math, record/skew/late statistics, histogram quantiles, fault scheduling, and full-job MiniCluster runs proving exact counts, late-data-without-loss, session grouping, skew→subtask concentration, rebalance evenness, in-operator fault-fires (SOURCE/AGGREGATE), and restart recovery. ~86% coverage.
  • Live on Docker Compose: only latency-soak has been driven end-to-end with real metrics + scorecard validated against live data. The other six are proven at MiniCluster + config level but not yet through the full compose stack.
  • Not yet exercised live: a real Runtime.halt() TM crash recovering on the dashboard, S3/MinIO checkpoint write+restore, and RAMP/BURST shapes on Grafana — all wired and unit/IT-tested, not visually confirmed.

Requirements: Docker (build/test/sim/helm) · a Flink K8s Operator 1.12 on each target cluster (Helm A/B) · Python 3.9+ (scorecard). No local Java/Maven/Flink needed.
